A human control plane for AI agents — where one person directs a hive of specialist agents that run locally or in the cloud, each keep their own memory, and route work to each other.

The system at a glance

One hive, five moving parts

Beevibe turns a single Claude Code CLI into a coordinated team. A thin web client talks to a stateless API; the API persists everything to Postgres and hands runnable work to executors — a local daemon on your machine or a server-side scheduler — which spawn real claude processes that call back through MCP for memory and coordination.

🖥️ Thin client

The web app holds no business logic — it queries the API, subscribes to a single SSE stream, and renders live transcripts as agents work.

🛰️ Stateless brain

The API composes prompts, assembles MCP tools per session, and writes to Postgres. The only in-memory state is the daemon WebSocket registry.

🤖 Distributed muscle

Daemon and scheduler are interchangeable executors. Both provision the same workspace and spawn the same CLI, so results are identical wherever they run.

async run(input: AgentSessionRunInput): Promise<Session> {
  // 1. Load the agent
  const agent = await this.deps.agentRepo.findById(input.agentId);

  // 2. Reuse the pending row the dispatcher inserted, or mint a new one
  const sid = input.sessionId ?? newSessionId();
  // ...create/update the session row at status 'running'...

  // 3. Build the per-session memory briefing + layered system prompt
  const briefing = await this.deps.memoryAgent.prepareBriefing(input.intent);
  const system_prompt_append = composeSystemPromptAppend(
    agent.runtime_config.system_prompt_addition,
    briefing.systemPromptAppend,
    { /* sessionKind, onboarding flags */ },
  );
  const intent = composeIntent(input.intent, briefing.userMessagePrefix);

  // 4. Execute the CLI — each RuntimeStep is teed into session_event
  const result = await this.deps.runtime.execute({
    intent, workspace: input.workspace, system_prompt_append,
    model: agent.runtime_config.model,
    max_turns: agent.runtime_config.max_turns,
    onStep,
  });

  // 5. Persist terminal state (status, usage, cli_session_id)
  const finalSession = await this.deps.sessionRepo.update(sid, { /* ... */ });

  // 6. Fire-and-forget: promote memory, then run completion hooks
  void this.deps.memoryAgent.onTaskComplete(sid);
  void this.deps.onSessionComplete?.(finalSession);
  return finalSession;
}

🔌 MCP router

Keeps a per-session map of { transport, server, beevibeSid }. On each call it assembles the tool set fresh over (caller, sessionId) — so handlers carry context with no async-local-storage threading. Tools write straight to the repos.

📡 DaemonHub

The one piece of in-memory state: a registry of connected daemon WebSockets keyed by runtime_id. When a session is inserted, the hub pushes a task_available frame so the daemon wakes instantly instead of waiting for its 30s poll. Tracks lastSeen to decide online vs. server-fallback.

‹/› Important code

▸ DaemonHub.notify() — WebSocket push that beats the poll api/src/runtime/hub.ts 
// Wake every daemon subscribed to runtimeId so it claims the pending
// session now, instead of waiting for the 30s HTTP claim poll.
notify(runtimeId: string, sessionId: string): void {
  const bucket = this.byRuntimeId.get(runtimeId);
  if (!bucket || bucket.size === 0) return;          // no daemon online
  const payload = {
    type: "task_available", runtime_id: runtimeId, session_id: sessionId,
  };
  for (const client of bucket) {
    // dedup'd per client so a double-notify fires only one wakeup
    this.deliver(client, `task_available:${sessionId}`, payload);
  }
}

🛰️ Daemon — the local runner

A lightweight process on your own machine. It detects installed CLIs, registers them with the API, then claims pending sessions and spawns claude locally — streaming every tool call back as it happens. This is how an agent can act on your filesystem with your Claude subscription.

📁 Workspace

Each agent owns ~/.beevibe/workspaces/<agent_id>/ — persistent repo clones, a generated mcp-config.json, and tier-filtered .claude/skills/. Survives across sessions so context compounds.

🔑 Identity

The daemon authenticates with its own bv_d_ token (argon2-hashed server-side), minted at setup and stored in ~/.beevibe/config.json alongside its registered runtimes.

🤖 Spawn shape

CLI runs with --dangerously-skip-permissions --strict-mcp-config, cwd = workspace, output parsed as stream-json. Env strips nesting guards & ANTHROPIC_API_KEY so it bills the subscription.

private async pollRuntime(runtimeId: string): Promise<void> {
  while (this.running && this.cfg.supervisor.hasCapacity()) {
    const payload = await this.cfg.api.claim<DispatchPayload>(runtimeId);
    if (!payload) return;                      // queue empty for this runtime
    const ctrl = this.cfg.supervisor.start(payload.session_id);

packages/scheduler

⏱️ Scheduler — server-side executor & safety net

For agents with no daemon bound, the scheduler is the fallback executor. It's a poll loop, not a cron table: every ~30s it reaps dead processes, claims ready sessions, and after each run decides what happens next — including nudging an agent that forgot to close its task.

🪦 Reapstale running + dead PID → fail & requeue
→
🎯 Claimatomic claimNextForServerFallback()
→
🚀 Dispatchprovision + AgentSession.run
→
🔁 Post-dispatchrollup · retry · nudge
📣 CancelListener — dedicated pg client on LISTEN cancel_task → aborts the in-flight subprocess
Polling executor with crash recovery and a post-dispatch decision tree.

🔁 Post-dispatch decision tree

  • Agent set a terminal status → roll the parent task up.
  • Non-terminal children still running → wait.
  • All children settled → run parent rollup.
  • Leaf finished but forgot update_progress → dispatch a retry session with a nudge.
  • Retry still silent after a grace window → fail the task.

📣 Cross-process cancel

The API's POST /task/:id/cancel fires pg_notify('cancel_task', id). The scheduler's listener — a dedicated pg client, not from the pool — receives it and aborts the matching AbortController, killing the CLI subprocess wherever the loop is.

‹/› Important code

▸ The poll loop — reap, then claim-while-capacity scheduler/src/worker.ts 
async poll(): Promise<void> {
  if (!this.running) return;
  await this.reapOrphanedSessions();   // dead PID + stale 'running' → fail & requeue
  await this.dispatchReady();
}

private async dispatchReady(): Promise<void> {
  while (this.running) {
    const session = await this.config.sessionRepo.claimNextForServerFallback();
    if (!session) break;                                  // nothing pending
    const agent = await this.config.agentRepo.findById(session.agent_id);
    if (!(await this.hasTaskCapacityForClaim(agent, session))) {
      await this.config.sessionRepo.update(session.id, { status: "pending" });
      break;                                              // over cap → release it back
    }
    const workspace = await this.config.workspaceManager.ensureWorkspace({ agent });
    void this.config.dispatchTask(session, agent, workspace, ac.signal);
  }
}

Two memory layers, per agent. core_memory_block is small, always in-context, and expensive per byte — persona, domain, constraints. memory_fact is unbounded, vector-indexed archival — beliefs, patterns, gotchas, decisions — retrieved top-k per session. Both are keyed by agent_id: every read filters WHERE agent_id = $self, so an agent only ever sees its own memory. The scope column (ic / team / org) and promotion only relabel an agent's own rows — they don't copy facts between agents.

▸ searchByVector() — memory reads are scoped to one agent core/src/adapters/postgres/memory-fact-repo.ts 
async searchByVector(params: VectorSearchParams): Promise<MemoryFact[]> {
  const { rows } = await this.pool.query<MemoryFactRow>(
    `SELECT ${FACT_COLUMNS},
            1 - (embedding <=> $1::vector) AS similarity
       FROM memory_fact
      WHERE agent_id = $2                      -- ← every read is scoped to ONE agent
        AND scope = ANY($3::text[])
        AND ($4::text[] IS NULL OR fact_type = ANY($4::text[]))
        AND (1 - (embedding <=> $1::vector)) >= $5
      ORDER BY embedding <=> $1::vector ASC     -- cosine distance over the HNSW index
      LIMIT $6`,
    [vecLit, params.agent_id, scopes, factTypes, minSim, limit],
  );
  return rows.map(rowToFact);
}

🐝 Agent hierarchy & routing

Agents form a three-tier tree. ICs are leaf specialists; team and org agents coordinate, delegate, and can spawn new specialists. A three-lane routing directive is injected into every team-agent session to decide who does the work.

🛣️ Lane A — do it yourself

Ambiguous scope, small fixes, cross-cutting coordination the team agent is best placed to handle directly.

📨 Lane B — delegate

Substantive single-domain work that matches an existing specialist. Routed via create_task to a subordinate.

✨ Lane C — spawn

A gap in the team. Propose a new IC via create_subordinate_agent — capped at 8 spawns / 24h, every one audited.

🧰 Tools assembled per session

On each request the router resolves the caller, loads the agent's hierarchy level, and builds the exact tool set it's allowed — IC vs. team — closed over (agentId, sessionId). Three families: memory, hierarchy, mesh.

Every tool call appends a session_event row (fire-and-forget). The X-Beevibe-Session header threads each HTTP call back to its session, so the transcript and SSE stream stay perfectly aligned.

putting it together

🔄 Lifecycles end to end

Three canonical paths show how the pieces interlock. Notice the shape repeats: dispatch a pending session → an executor claims it → the CLI runs and calls MCP → results persist → hooks fire → the UI updates over SSE.

① A chat message

  1. web User sends a message; the client pre-mints a session_id and subscribes to its SSE step stream.
  2. api /chat creates a running session, composes the chat prompt stack, and runs the agent's CLI.
  3. cli → mcp The agent reads context, calls memory/hierarchy tools; each call streams back as a session.step event.
  4. api → web On completion the reply returns with view refs; the live transcript has already rendered each step.

② A task on a daemon-bound agent

  1. web User creates a task assigned to an agent with a preferred_runtime_id.
  2. api DispatchService inserts a pending session and DaemonHub pushes task_available over WebSocket.
  3. daemon The Claimer wakes, calls /runtime/claim, gets the briefing, provisions the workspace, spawns claude.
  4. daemon → api Steps stream to /runtime/events (buffered ~250ms); fanned out as SSE to the web UI.
  5. daemon → api CLI exits → POST /runtime/done writes status, usage, cli_session_id.
  6. api onSessionComplete fires: memory promotion, mesh resolver, task rollup; SSE tells the UI it's done.

③ The same task with no daemon (server fallback)

  1. api Session is inserted pending; no daemon is online for the runtime.
  2. scheduler The poll loop reaps orphans, then atomically claims the pending session.
  3. scheduler It provisions the identical workspace and runs AgentSession.run() in-process.
  4. scheduler Post-dispatch decides next: roll up the parent, retry with a nudge, or fail after the grace window.
Why two executors? The daemon lets an agent act on your machine with your subscription; the scheduler guarantees work still runs when no daemon is connected. Both call the same core service against the same workspace shape — so a task's outcome doesn't depend on where it ran.
